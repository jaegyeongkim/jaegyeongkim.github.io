모노레포 전환을 처음 제안했을 때 팀 반응은 "복잡해지는 거 아니냐"였다. 지금은 "예전에 어떻게 개발했지?"라고 한다.

전환 전 상황이 어땠냐면, 10개 앱이 독립된 저장소에 따로 있었다. `Button`, `Table`, `Input` 같은 공용 컴포넌트가 앱마다 조금씩 다른 버전으로 복사돼 있었다. 버그를 한 곳에서 고쳐도 나머지 9개에는 반영이 안 됐다. 디자인 변경이 생기면 각 앱을 열어서 따로 수정해야 했다.

이 구조가 언제까지 버틸 수 없겠다 싶었던 계기는 TypeScript 마이그레이션이었다. JavaScript → TypeScript로 전환하는 작업을 10개 앱에서 각자 해야 하는 상황이 됐는데, 그때 "이건 한 번에 끝내야 한다"고 확신했다.

## 도구 선택

Lerna, Nx, Turborepo를 검토했다.

Lerna는 예전부터 있던 도구인데, 빌드 캐싱이 약하다는 게 걸렸다. Nx는 기능이 많지만 설정이 복잡하고 팀 규모 대비 오버스펙이었다. Turborepo는 파이프라인 설정이 단순하고 pnpm workspace와 잘 맞아서 선택했다.

pnpm을 고른 이유는 **phantom dependency 문제** 때문이다. npm과 yarn은 `node_modules` hoisting 때문에 직접 설치하지 않은 패키지를 `import`할 수 있다. 어느 날 의존성 버전이 바뀌면서 조용히 깨지는 그런 문제가 생긴다. pnpm은 symlink 구조로 이를 막는다. 공용 패키지를 10개 앱이 함께 쓰는 환경에서는 이게 중요했다.

## 패키지 구조

```
packages/
├── components/    # 공용 UI 컴포넌트 206개+
├── stores/        # 공용 Zustand 스토어
├── hooks/         # 공용 훅
├── types/         # 공용 TypeScript 타입
├── utils/         # 공용 유틸리티
├── schemas/       # Zod 스키마
├── assets/        # 정적 리소스 + 설정 상수
└── styles/        # 공용 스타일
```

초기에는 `packages/components`에 모든 걸 다 넣었다가, 규모가 커지면서 목적별로 분리했다. `utils`에 있던 함수가 나중에 보니 `hooks`여야 했던 경우도 있었고, 경계가 모호한 것들은 팀에서 그때그때 논의해서 정했다.

## 빌드 순서와 캐싱

Turborepo에서 중요한 건 태스크 의존성 선언이다.

```json
{
  "tasks": {
    "build": {
      "dependsOn": ["^build"],
      "outputs": [".built-packages/**", "dist/**"]
    },
    "dev": {
      "dependsOn": ["^build"],
      "persistent": true
    }
  }
}
```

`^build`는 "내가 의존하는 패키지의 build가 먼저 완료돼야 한다"는 뜻이다. 앱이 패키지를 참조하는데 패키지 빌드가 안 돼 있으면 개발 서버가 이전 빌드 결과물을 보거나 에러가 난다. 이 선언 하나로 순서가 자동으로 보장된다.

빌드 캐싱도 있다. 파일이 바뀌지 않은 패키지는 이전 결과를 재사용한다. CI에서 `packages/components`를 건드리지 않은 PR은 컴포넌트 빌드를 통째로 건너뛴다.

## 테스트가 없으면 의미가 없다

공용 컴포넌트 206개를 운영하는데 테스트가 없으면, 수정이 두렵다. Vitest + Testing Library로 컴포넌트마다 테스트를 붙였다. 지금 165개 테스트를 운영 중이다.

Zustand 스토어를 쓰는 컴포넌트를 테스트할 때는 스토어 mock을 selector 인터페이스 그대로 구현한다.

```ts
vi.mock("@repo/stores/queryFilterState", () => ({
  useQueryFilterStore: (selector) => selector(mockUseQueryFilterStore),
}));
```

실제 스토어를 띄우지 않고도 컴포넌트 동작을 정확하게 검증할 수 있다. PR에서 패키지가 변경되면 GitHub Actions가 해당 패키지 테스트를 돌린다. 실패하면 merge가 안 된다.

---

공용 컴포넌트가 하나 바뀔 때 "다른 앱 안 깨지겠지?" 하고 불안해하지 않아도 된다는 게 지금은 당연해 보이지만, 전환 전에는 그게 당연하지 않았다. 테스트가 통과하면 바꿔도 된다. 이 한 문장이 생산성에서 꽤 큰 차이를 만든다.
