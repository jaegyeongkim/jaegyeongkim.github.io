# 블로그 글쓰기 가이드

이 디렉토리(`content/blog/*.mdx`)에 **새로 쓰는 글**은 아래 흐름과 문체를 따른다. 올리브영 테크 블로그("이 버튼 왜 안 눌려요?", "배송최적화 시스템 구축기") 스타일을 참고했다.

> ⚠️ 기존에 이미 작성된 글(`typia-runtime-validation.mdx` 등 기존 8개)은 이 가이드가 나오기 전 다른 문체(1인칭 반말 회고체)로 쓰였다. **기존 글은 이 가이드에 맞춰 고치지 않는다.** 이 가이드는 새로 추가되는 글에만 적용한다.

## 글의 구조: 반복되는 "해결 사이클"

문제 → 고민 → 원인 → 해결 → 트레이드오프 → 적용 중 어려움 → 검증 → 결과, 이 8가지 요소를 **한 번에 순서대로 다 채우는 게 아니다.** 실제 올리브영 글은 아래처럼 큰 틀 안에서 "해결 사이클"을 여러 번 반복하는 구조를 쓴다.

```
1. 도입부 — 장면 묘사로 시작 (아래 "오프닝" 참고)
2. 배경 / 제약사항 — 왜 이 문제를 풀어야 했는지 맥락
3. 기술/방향 선택 고민 — 후보안 비교, 보통 표 1개로 정리 (전체 글에서 1번)
4. 해결 사이클 (N차 해결, 필요한 만큼 반복)
   - 이번 사이클에서 부딪힌 문제
   - 원인 (왜 안 됐는지)
   - 해결 방법 (코드/설정/구조)
   - 적용 중 예상 못한 문제가 있었다면 → 그게 다음 사이클로 이어짐
5. 종합 결과 — 글 전체 결과를 맨 끝에 수치로 몰아서 제시
6. 배운 점 / 교훈 — 이 프로젝트에서만 나올 수 있는 구체적 교훈 2~4개
```

- "기술/방향 선택 고민"과 "트레이드오프"는 개별 해결 사이클마다 반복하지 않고, 글 초반 **한 번 크게** 다룬다 (예: Native App vs Web App 비교표).
- "적용 후 검증"은 별도 섹션이 아니라, 배포 후 발견된 문제를 다루는 **또 하나의 해결 사이클**로 자연스럽게 이어지게 쓴다 (예: "현장 테스트로 발견한 키패드 이슈" = 3차 해결).
- 소제목은 8가지 요소 이름을 그대로 쓰지 않는다. "N차 해결: {핵심 개념}"처럼 그 사이클에서 실제로 도입한 기술/개념 이름을 쓴다.

## 오프닝 (도입부) 작성법

첫 문단은 절대 "이 글에서는 ~를 다룹니다" 같은 설명으로 시작하지 않는다. 아래 중 하나로 연다.

- **현장/장면 묘사**: 의성어, 구체적 상황 묘사로 시작 ("삐빅, 삐빅." 1분 1초가 급박하게 돌아가는 물류센터...)
- **질문형 문장 반복**: "~라면? ~라면?" 식으로 문제 상황을 독자가 상상하게 만든 뒤 "실제로 저희가 마주한 현실이었습니다"로 매듭짓는다.
- **수치/사실 기반 배경**: 회사·서비스의 성장이나 규모 변화로 시작해 "그런데 이 성장이 오히려 새로운 문제를 낳았다"로 전환한다.

도입부 마지막에는 이 글이 다루는 내용을 1~2문장으로 미리 요약하고, "비슷한 고민을 하는 개발팀에게 도움이 되길 바란다" 톤으로 짧게 마무리한다.

## 문체

- **존댓말**로 쓴다 ("~했습니다", "~입니다"). 기존 글의 반말 회고체("~했다")를 따라하지 않는다.
- 원인/장점/설계 원칙처럼 여러 개를 나열할 때는 **bold로 핵심 문구를 소제목처럼 강조**한 뒤 설명을 붙이는 리스트를 쓴다.
  ```
  - **데이터 단절로 인한 판단 불가**: 주문 단계에서 실시간 재고나 센터별 가동률 같은 물류 데이터를 즉시 확인하기 어려웠습니다.
  ```
- 이모지는 소량만 쓴다. 섹션 전환이나 감정이 실리는 지점에만 (🚀 🥲 🎯 👉 🔗 💡). 문장마다 붙이지 않는다.
- 💡 콜아웃 형태로 "프로세스 요약"이나 "참고"를 별도로 뽑아준다.
  ```
  💡 최적의 센터를 찾아가는 프로세스
  1단계. ...
  2단계. ...
  ```
- 표는 기술/대안 비교에 적극 활용한다. 표 없이 장단점을 줄글로 풀어쓰지 않는다.
- 정량적 결과는 문장 속에 흘려쓰지 않고, 섹션 끝에 강조해서 모아 보여준다 (가능하면 "지표 43.5% ↑" 같은 카드/불릿 형태).
- 결과가 수치로 안 나오는 항목은 지어내지 말고 정성적 변화(예: "불편하다는 피드백이 사라졌습니다")로 쓴다.
- 이 레포는 이미지 첨부 파이프라인이 없으므로 이미지·스크린샷은 넣지 않는다. 원본 올리브영 글의 이미지 자리는 표/코드블록/💡 박스로 대체한다.

## 마무리 작성법

마지막 섹션은 다음 요소를 순서대로 포함한다.

1. 종합 결과 수치 요약
2. 배운 점 / 교훈 (일반론 금지, 이 프로젝트에서만 나올 수 있는 구체적 교훈)
3. (선택) 앞으로의 계획이나 로드맵
4. (선택) 독자에게 질문을 던지며 마무리 ("여러분의 서비스도 비슷한 고민을 앞두고 계신가요?")

## 체크리스트

새 글을 쓴 뒤 아래를 확인한다.

- [ ] 도입부가 설명이 아니라 장면/질문형 문장으로 시작하는가
- [ ] 존댓말로 일관되게 쓰였는가 (반말 섞이지 않았는가)
- [ ] "기술/방향 선택 고민"이 표로 한 번 정리되어 있는가
- [ ] 해결 사이클(N차 해결)마다 문제-원인-해결이 드러나는가, 트레이드오프를 매번 반복해서 늘어놓지 않았는가
- [ ] 원인/장점 나열에 bold 강조 리스트를 썼는가
- [ ] 결과가 마지막에 수치로 몰아서 정리되어 있는가 (지어낸 수치 없이)
- [ ] 마무리에 교훈이 구체적으로 들어가 있는가
- [ ] `lib/blog-posts.ts`의 `BLOG_POSTS`에 slug/title/description/date/tags를 추가했는가
