"대용량 데이터를 다뤄보신 적 있으신가요?"

면접 자리에서 이 질문을 몇 번이나 받았습니다. 그때마다 선뜻 "네"라고 답하기가 어려웠습니다. 제가 다뤄본 데이터는 수십만 건 단위의 배치 처리도, 실시간 스트리밍도 아니었기 때문입니다.

그런데 같은 질문을 반복해서 받다 보니, 면접관이 정말 궁금해하는 건 다뤄본 데이터의 '규모' 자체가 아니라는 걸 알게 됐습니다. 오히려 "그 데이터를 화면에 그리다가 어떤 문제를 만났는지, 그 문제를 어떻게 분석했는지, 그래서 어떻게 풀었는지"를 묻고 있었습니다. 프론트엔드 입장에서는 데이터가 얼마나 큰지보다, 그 데이터를 렌더링하는 순간 무엇이 느려지고 무엇이 깨지는지가 진짜 문제니까요.

돌아보니 답할 수 있는 경험이 하나 있었습니다. moveRanking 앱의 랭킹 화면에서, 드라이버 수가 무제한으로 늘어날 수 있는 리스트를 어떻게 그릴지 고민했던 이야기입니다. 이 글에서는 그 과정에서 시도했다가 롤백한 접근과, 최종적으로 선택한 해법을 공유합니다. 비슷한 고민을 하고 계신 분들께 도움이 되길 바랍니다.

## 무제한으로 늘어나는 리스트, 어떻게 그릴 것인가

moveRanking의 랭킹 화면은 드라이버 수가 n명 이상으로 무제한 늘어날 수 있는 구조였습니다. 그런데 `List.tsx`는 `data.drivers` 전체를 `.map()`으로 순회해, 화면에 보이지 않는 행까지 전부 DOM 노드로 만들고 있었습니다. 드라이버 수가 늘어날수록 초기 렌더링 부하도 그만큼 비례해서 커지는 구조였던 셈입니다. 게다가 랭킹 데이터를 내려주는 `GET /ranking` API가 페이지네이션 없이 전체 목록을 한 번에 응답하는 구조라, 백엔드 변경 없이 프론트 단독으로 풀어야 한다는 제약도 있었습니다.

여기에 지켜야 할 기존 기능이 하나 더 있었습니다. queryParameter에 `driverId` 값이 있으면 "내 순위로 이동" 버튼을 눌렀을 때 해당 랭킹으로 바로 스크롤 이동하는 기능이었는데, 렌더링 방식을 바꾸는 김에 이 기능까지 함께 깨지면 안 됐습니다.

원칙은 명확했습니다. **백엔드 API를 바꾸지 않고, 화면에 필요한 만큼만 그리는 방향으로 풀자.** 이 원칙을 구현할 방법으로 두 가지를 검토했습니다.

| 방식                     | 동작 방식                                                  | 장점                                                    | 단점                                                                                       |
| ------------------------ | ---------------------------------------------------------- | ------------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| 가상 스크롤 (windowing)  | 뷰포트 근처 행만 마운트하고, 스크롤에 따라 DOM 노드를 교체 | 스크롤을 끝까지 내려도 마운트된 DOM 노드 수가 항상 일정 | `position: absolute` + `transform` + `scrollMargin`을 수동으로 계산해야 해 구조가 복잡해짐 |
| 점진적 렌더링 (chunking) | 처음 N개만 렌더링하고, Sentinel에 도달하면 개수를 늘림     | 기존 `.map()` 구조를 그대로 활용해 구현·디버깅이 단순   | 스크롤을 끝까지 내리면 렌더링된 DOM 노드 수가 계속 누적됨                                  |

메모리·DOM 노드 수 측면에서는 "진짜" windowing 쪽이 이론적으로 더 우아한 해법이라고 판단해, 먼저 가상 스크롤을 선택했습니다.

## 1차 문제: 가상 스크롤 도입 후 원인을 특정할 수 없는 렌더링 간격 버그

`@tanstack/react-virtual`의 `useWindowVirtualizer`로 뷰포트 근처 행만 마운트하도록 `List.tsx`를 리팩토링했습니다. 그런데 화면에서 행 사이 간격이 비정상적으로 벌어지는 버그가 발견됐습니다.

원인을 특정하기 위해 할 수 있는 검증은 다 해봤습니다.

- 라이브러리 소스를 직접 재현해가며 동작을 추적
- `getComputedStyle` 콘솔 검증 — `translateY` 값이 92px 간격으로 정확히 계산되고 있음을 확인
- 스크린샷과 실제 렌더링 화면을 나란히 대조
- 스크롤·리사이즈로 강제 리페인트를 유도해 재현 시도

그런데 DOM/CSS 값은 항상 정상인데, 실제 화면은 계속 어긋나는 현상만 재현될 뿐이었습니다. 모바일 실기기 원격 디버깅 환경에서만 나타나는 문제인지, 특정 브라우저·특정 검증 조합(스크린샷과 콘솔을 동시에 띄운 환경 등)에서만 벌어지는 문제인지조차 끝내 구분하지 못한 채, 근본 원인은 특정되지 않은 상태로 남았습니다.

- **판단**: 원인 자체를 모르는 상태에서, `position: absolute` + `transform` + `scrollMargin`을 수동으로 계산하는 조합을 계속 붙들고 있는 건 디버깅 난이도와 리스크만 키운다고 판단했습니다. 원인 불명 상태로 프로덕션에 내보내기에는 위험 부담이 컸습니다.

## 2차 문제: 청크 렌더링으로 바꾸면 "내 순위로 이동" 기능이 깨진다

가상 스크롤을 포기하고 청크 렌더링(처음 50명만 표시)으로 전환하기 전에, 기존 기능에 영향이 없는지부터 점검했습니다. 그 결과 처음 50명만 렌더링하는 방식으로 바꾸면, `driverId`가 가리키는 드라이버가 아직 렌더링되지 않은 범위 밖에 있을 경우 이동할 DOM 노드 자체가 없어 "내 순위로 이동" 기능이 그대로는 동작하지 않는다는 걸 확인했습니다.

- **해결**: 가상 스크롤 구현은 `git stash`로 백업해두고, 원본 코드로 롤백한 뒤 다음과 같이 재구성했습니다.
  - `Layer.tsx`가 `visibleCount` 상태를 소유합니다 (초기값 50, `IntersectionObserver`가 하단 도달을 감지하면 +50).
  - `List.tsx`는 기존 `.map()` 구조를 그대로 유지한 채 `data.drivers.slice(0, visibleCount)`만 렌더링하고, 리스트 하단에 보이지 않는 Sentinel 요소를 추가 로딩 트리거로 둡니다.
  - `ButtonField.tsx`가 `searchParams`에서 `driverId`를 읽어 `data.drivers`에서 몇 번째 순위인지(`rank`)를 미리 계산해 `useButtonField`에 넘깁니다.
  - `useButtonField.ts`의 `handleRectangleButtonClick`이 이 `rank`에서 위쪽 여유를 위한 오프셋(`position = 4`)만큼 뺀 인덱스를 스크롤 대상으로 삼고, `scrollToIndex`가 ref 존재 여부를 먼저 확인한 뒤 없으면 `ensureVisible`을 호출하는 2단계 로직으로 "내 순위로 이동" 기능을 유지했습니다.
  - `ListItem.tsx`는 변경 없이 원본 ref 콜백 방식을 그대로 재사용합니다.

```ts
// useButtonField.ts (실제 구현에서 핵심 로직만 발췌)
const pendingScrollIndexRef = useRef<number | null>(null);

const isRankingListRefsExist = (index: number) =>
  rankingListRefs.current && rankingListRefs.current[index];

const scrollToIndex = (index: number): void => {
  if (isRankingListRefsExist(index)) {
    rankingListRefs.current[index]!.scrollIntoView({
      behavior: "smooth",
      block: "start",
    });
    return;
  }

  // 아직 마운트되지 않은 범위 → visibleCount를 확장하고 인덱스만 예약
  pendingScrollIndexRef.current = index;
  ensureVisible(index);
};

const handleRectangleButtonClick = (rank: number) => (): void => {
  const position = 4; // 위쪽 여유를 위한 오프셋

  return rank - position >= position
    ? scrollToIndex(rank - position)
    : handleTop10ButtonClick();
};

// 예약된 인덱스가 실제로 마운트되는 시점(visibleCount 변경)에 이동
useEffect(() => {
  const pendingIndex = pendingScrollIndexRef.current;
  if (pendingIndex === null || !isRankingListRefsExist(pendingIndex)) return;

  rankingListRefs.current[pendingIndex]!.scrollIntoView({
    behavior: "smooth",
    block: "start",
  });
  pendingScrollIndexRef.current = null;
}, [visibleCount, rankingListRefs]);
```

💡 "내 순위로 이동" 2단계 로직
1단계. `scrollToIndex`가 먼저 `rankingListRefs`에 해당 인덱스의 ref가 이미 있는지 확인합니다. 이미 렌더링된 범위라면 바로 `scrollIntoView`를 호출합니다.
2단계. ref가 아직 없다면(= 아직 렌더링되지 않은 범위) 인덱스를 `pendingScrollIndexRef`에 저장하고 `ensureVisible(index)`로 `visibleCount`를 확장합니다. 이후 `visibleCount` 변경을 감지하는 `useEffect`에서 ref가 생긴 걸 확인하고, 그제서야 `scrollIntoView`를 실행합니다.

마지막으로 `@tanstack/react-virtual` 의존성과 `pnpm-lock.yaml`에 남아있던 관련 항목을 완전히 제거했습니다.

## 종합 결과

- `position: absolute` / `transform` / `scrollMargin` 계산이 전혀 없는 구조라, 이번에 겪은 것과 같은 원인 불명 렌더링 버그가 발생할 여지 자체가 사라졌습니다.
- 초기 렌더링이 50개로 제한돼, 드라이버 수가 늘어나도 초기 렌더링 부하가 더 이상 비례해서 커지지 않습니다.
- `driverId` 쿼리파라미터 기반 "내 순위로 이동" 기능도 청크 밖 순위까지 정상 동작합니다.
- `tsc --noEmit`, `eslint`, `vite build` 모두 통과를 확인했습니다.

다만 진짜 windowing과 달리, 스크롤을 끝까지 내리면 렌더링된 DOM 노드 수가 계속 누적되는 트레이드오프는 그대로 남았습니다. 실사용 시나리오상 초기 렌더링 부하 감소가 핵심 목적이었기 때문에, 이 정도는 허용 가능한 트레이드오프라고 판단했습니다.

## 배운 점

- **기존 기능은 새 구현을 시작하기 전에 영향 범위부터 확인해야 합니다.** 청크 렌더링으로 전환하기 전에 "`driverId` 기반 이동 기능이 깨지지 않는가"를 미리 점검했기 때문에, `ensureVisible` + `pendingScrollIndexRef`라는 2단계 로직을 처음부터 설계에 포함시킬 수 있었습니다. 만약 청크 렌더링을 먼저 구현하고 나서 이 문제를 뒤늦게 발견했다면, 또 한 번 롤백에 가까운 리팩토링이 필요했을 겁니다.
- **원인을 특정할 수 없다면, 원인 규명보다 리스크 회피가 더 합리적인 결정일 수 있습니다.** 가상 스크롤 버그는 `getComputedStyle` 검증까지 마쳤는데도 DOM/CSS 값과 실제 화면이 계속 어긋나는, 재현은 되지만 설명은 안 되는 현상이었습니다. 원인을 끝까지 파고드는 대신 `position: absolute` + `transform` + `scrollMargin` 조합 자체의 디버깅 난이도를 근거로 롤백을 선택한 게, 결과적으로 더 빠르게 문제를 없애는 길이었습니다.
- **"더 우아한 해법"이 항상 "더 안전한 해법"은 아닙니다.** 가상 스크롤은 DOM 노드 수 측면에서 이론적으로는 더 나은 해법이었지만, 좌표를 수동으로 계산하는 구조 자체가 디버깅 리스크를 함께 끌고 들어왔습니다. 반면 청크 렌더링은 기존 `.map()` 구조와 ref 콜백을 그대로 재사용할 수 있어, 새로운 버그가 끼어들 여지 자체가 적었습니다.

## 남은 이슈: 실 API 환경에서의 최종 검증

moveRanking은 dev-mock을 지원하지 않는 앱이라, 로컬에서는 무한 스크롤과 "내 순위로 이동" 버튼의 동작을 끝까지 확인하기 어려웠습니다. 실 API·스테이징 환경에서 무한 스크롤 트리거와 순위 이동 버튼이 실제 데이터로도 정상 동작하는지 최종 브라우저 검증이 아직 남아 있습니다.

여러분의 서비스도 백엔드 페이지네이션 없이 프론트에서 대용량 리스트를 다뤄야 하는 제약을 안고 계신가요? 그렇다면 더 정교한 해법을 먼저 찾기보다, 지켜야 할 기존 기능부터 먼저 점검해보시길 권합니다.
