"이 버그, 저희 앱에서도 나는데요?" 슬랙에 이 메시지가 올라오면, 저희 팀은 반사적으로 긴장했습니다. 같은 `Button`, `Table`, `Input` 컴포넌트가 10개 앱에 각각 복사돼 있었기 때문에, 한 곳에서 고친 버그가 나머지 9곳엔 반영되지 않는 게 일상이었습니다. 디자인이 바뀌면 앱 10개를 하나씩 열어서 똑같은 수정을 반복해야 했습니다.

이 구조가 더는 못 버틴다고 확신한 계기는 JavaScript → TypeScript 마이그레이션이었습니다. 같은 작업을 10개 레포에서 각자 따로 해야 하는 상황 앞에서, "이건 한 번에 끝내야 한다"는 게 팀 전체의 결론이었습니다.

## 왜 Turborepo였나

Lerna, Nx, Turborepo를 검토했습니다. Lerna는 오래된 도구인 만큼 빌드 캐싱이 약하다는 점이 걸렸고, Nx는 기능은 풍부했지만 팀 규모 대비 설정이 무거웠습니다. Turborepo는 파이프라인 설정이 단순하고, 이미 쓰고 있던 pnpm workspace와 자연스럽게 맞물려서 선택했습니다.

pnpm을 쓰는 이유는 **phantom dependency 문제** 때문이었습니다. npm/yarn은 `node_modules` hoisting 구조 때문에 `package.json`에 직접 선언하지 않은 패키지도 `import`가 되어버립니다. 어느 날 다른 패키지의 버전이 바뀌면서 조용히 깨지는 사고가 나기 쉬운 구조입니다. pnpm은 symlink 기반이라 이런 "우연히 되는" 상황을 원천 차단합니다. 10개 앱이 공용 패키지를 함께 쓰는 환경에서는 이 차이가 결정적이었습니다.

## 1차 해결: 일단 되게 만들기

`npm install turbo --global`로 시작해, 기존 `node_modules`와 `package-lock.json`을 지우고 pnpm으로 재설치했습니다. 앱마다 제각각이던 스크립트 명령어를 정리하고 `.gitignore`도 손봤습니다. 이 단계는 최소한의 PoC였고, 검증이 끝난 뒤 6개 프로젝트를 한 번에 마이그레이션하는 본작업으로 넘어갔습니다.

## 2차 해결: 마이그레이션 직후 터진 자잘한 문제들

6개 프로젝트를 옮기고 나니, 크지는 않지만 무시할 수 없는 문제들이 연달아 나타났습니다.

- **CJS 경고**: 터미널에 계속 CJS 관련 경고 메시지가 떴습니다. 설정 파일 확장자를 Turborepo 권장 방식에 맞게 정리해 해결했습니다.
- **캐시 파일이 엉뚱한 곳에 쌓임**: Vite의 임시 캐시 파일(`vite.config.*.timestamp`)이 예상 밖의 위치에 생성돼 저장소를 어지럽혔습니다. 생성 위치를 재설정했습니다.
- **import 경로가 앱마다 제각각**: `packages`의 코드를 가져올 때 어떤 곳은 `@`, 어떤 곳은 상대 경로를 썼습니다. `apps`에서는 Turborepo가 권장하는 `@repo/*` 표기로 통일하고, `packages` 내부(패키지 간 참조)는 기존 `@` 표기를 유지하기로 정리했습니다. 전부 `@repo`로 밀어버리지 않고 경계를 둔 이유는, 패키지 내부 참조까지 `@repo`로 바꾸면 패키지 자신을 외부 패키지처럼 참조하는 꼴이 되어 오히려 헷갈렸기 때문입니다.

## 3차 해결: 캐시가 못 믿을 캐시였다

가장 뼈아팠던 문제는 따로 있었습니다. `turbo.json`의 `inputs`, `dependsOn` 설정이 허술해서, `packages` 폴더의 코드가 바뀌거나 `.env` 파일이 바뀌어도 Turborepo가 이를 감지하지 못하고 이전 빌드 캐시를 그대로 재사용하는 경우가 있었습니다. 캐시가 오래된 결과물을 계속 내놓으면, 캐시는 있으나 마나 한 게 아니라 오히려 없는 것보다 위험합니다. `inputs`에 감지 대상 파일을, `dependsOn`에 태스크 간 순서를 명시적으로 다시 적어 이 문제를 없앴습니다.

여기서 한 걸음 더 들어가, `packages/*` 각각의 `package.json`에 의존성이 제대로 선언되지 않은 것도 발견했습니다. 루트 `package.json`에 얹혀 "어쩌다 보니" 동작하고 있었던 겁니다. 이걸 그대로 두면 안 되는 이유가 세 가지 있었습니다.

- **패키지 독립성이 깨진다**: 어떤 라이브러리에 실제로 의존하는지 그 패키지의 `package.json`만 보고는 알 수 없었습니다.
- **pnpm은 hoisting을 최소화한다**: npm/yarn과 달리 pnpm은 선언한 의존성만 연결해줍니다. 선언하지 않은 패키지는 지금 우연히 되더라도, CI나 다른 개발자 환경, 혹은 버전이 바뀌는 순간 resolve에 실패할 수 있었습니다.
- **Turborepo의 캐시가 의존성 그래프를 기준으로 동작한다**: 패키지가 실제 의존성을 정확히 선언해야 캐시 무효화 범위가 정확해집니다. 선언이 허술하면 캐시가 다시 부정확해지는, 앞서 고친 문제로 되돌아가는 셈이었습니다.

전 패키지의 `dependencies`, `devDependencies`를 실제 사용 현황에 맞춰 일괄 재작성했습니다.

## 종합 결과

- 10개 앱이 동일한 공용 패키지를 사용 — 한 곳을 고치면 즉시 전체에 반영
- 관리해야 할 저장소 수가 10개에서 1개로 줄어 레포 간 동기화 작업 자체가 사라짐
- `hooks` 46개 중 44개, `utils` 40개 중 37개, `components` 164개 중 72개에 Vitest 테스트 적용 — 공용 코드 라인 커버리지 52%

## 배운 점

- **모노레포 전환은 "옮기는" 순간보다 "옮긴 다음"이 더 깁니다.** import 컨벤션 통일, 캐시 설정, 의존성 명시화까지 마이그레이션 이후에 발견되는 문제가 더 많았습니다.
- **캐시는 정확하지 않으면 부채입니다.** `inputs`/`dependsOn`을 대충 적어두면, 캐시가 있다는 사실 자체가 "최신 코드가 아닐 수도 있다"는 의심으로 바뀝니다.
- **"우연히 동작한다"는 pnpm 모노레포에서 제일 먼저 없애야 할 상태입니다.** 루트에 얹혀서 되던 의존성은 언젠가 반드시 문제를 일으킵니다. 발견했을 때 바로 명시적으로 고쳐야, 나중에 원인 모를 CI 실패를 디버깅하지 않게 됩니다.
