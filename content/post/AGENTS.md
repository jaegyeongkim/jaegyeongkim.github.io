# 포스트 글쓰기 가이드

이 디렉토리(`content/post/*.mdx`)에 **새로 쓰는 글**은 아래 흐름과 문체를 따른다. 올리브영 테크 블로그("이 버튼 왜 안 눌려요?", "배송최적화 시스템 구축기") 스타일을 참고했다.

> ⚠️ `typia-runtime-validation.mdx`를 제외한 기존 글들은 이 가이드가 나오기 전 다른 문체(1인칭 반말 회고체)로 쓰였다. **그 기존 글들은 이 가이드에 맞춰 고치지 않는다.** 이 가이드는 새로 추가되는 글과, 이미 이 가이드에 맞춰 다듬은 `typia-runtime-validation.mdx`에 적용한다.

## 글의 구조: 8단계 흐름을 3개 섹션에 담기

내용상 지켜야 할 순서는 8단계지만, 실제 올리브영 글은 이 8단계를 8개의 소제목으로 쪼개지 않는다. **문제 → 결정 → 비교와 선택**은 하나의 "배경" 섹션 안에서 문단으로 자연스럽게 이어지고, **어려움 → 해결**은 "N차 문제" 사이클로 반복되며, **결과 → 회고 → 남은 이슈**는 마지막 섹션에 몰린다.

```
[배경 섹션] (소제목 없이 오프닝에 바로 이어지거나, "왜 이 문제를 풀어야 했나" 류의 소제목 1개)
  1. 문제        — 어떤 문제가 있었는지 (오프닝: 장면 묘사로 이미 다룸)
  2. 결정        — 그 문제를 어떤 원칙/방향으로 풀기로 했는지
                   (아직 특정 기술 이름이 아니라 팀이 세운 원칙 —
                    예: "API 값을 무조건 신뢰하지 않고 검증하는 계층을 두기로 했다")
  3. 비교와 선택 — 그 원칙을 구현할 방법으로 무엇을 비교했고,
                   이런 이유로 무엇을 선택했는지 (보통 표 1개, 전체 글에서 1번만)

[N차 문제] (필요한 만큼 반복, 각 사이클이 소제목 하나)
  4. 어려움 — 이번 사이클에서 부딪힌 문제 (Before 수치가 있다면 여기서 언급)
  5. 해결   — 어떻게 해결해나갔는지 (코드/설정/구조)
             예상 못한 문제가 또 나왔다면 → 다음 사이클로 자연스럽게 이어짐
  * Before/After 비교표는 4~5 이후, 사이클 맨 끝에 결과로 배치한다.
    "문제를 재보니 이랬다" 표를 먼저 던지고 원인·해결을 뒤에 붙이지 않는다 —
    표는 해결까지 다 설명한 다음 "그래서 이만큼 좋아졌다"는 페이백으로 온다.

[마무리 섹션] (소제목 1개, 보통 "종합 결과" 류)
  6. 결과      — 글 전체 결과를 수치로 몰아서 제시
  7. 회고      — 이 프로젝트에서만 나올 수 있는 구체적 배운 점 2~4개
  8. 남은 이슈 — (있다면) 아직 못 푼 문제, 어떻게 푸는 게 옳다고 생각하는지
```

- **2(결정)에서 3(비교와 선택)으로 건너뛸 때 특정 기술 이름을 먼저 말하지 않는다.** 2는 "우리는 이런 원칙으로 가기로 했다"는 팀의 결론까지만 말하고, 그다음 문단에서 비로소 "이 원칙을 구현할 방법을 비교해보니…"로 넘어가 3을 다룬다. "그러면 Typia를 썼습니다"처럼 2에서 곧장 특정 기술명으로 끝내지 않는다.
- 3(비교와 선택)과 트레이드오프는 개별 사이클마다 반복하지 않고, 배경 섹션에서 **한 번 크게** 다룬다 (예: Native App vs Web App 비교표, yup vs Typia 성능 비교표).
- "적용 후 검증"은 별도 섹션이 아니라, 배포 후 발견된 문제를 다루는 **또 하나의 N차 문제**로 자연스럽게 이어지게 쓴다 (예: "현장 테스트로 발견한 키패드 이슈" = 3차 문제).
- 소제목은 "N차 해결"이 아니라 **"N차 문제: {실제로 터진 문제}"**로 쓴다. 그 사이클의 본문이 해결책을 설명하더라도, 소제목은 사이클을 열게 만든 문제 쪽에 맞춘다. 소제목만 보고도 "이번엔 무슨 문제가 있었는지"가 바로 보여야 한다.
- **비난·저격 톤을 피한다.** 원인이 다른 팀·다른 시스템에 있었더라도 "잘못은 아니었다", "눈총을 받았다" 같은 감정적 표현 대신, 사실 관계만 담백하게 서술한다.

## 오프닝 (도입부) 작성법

첫 문단은 절대 "이 글에서는 ~를 다룹니다" 같은 설명으로 시작하지 않는다. 아래 중 하나로 연다.

- **현장/장면 묘사**: 의성어, 구체적 상황 묘사로 시작 ("삐빅, 삐빅." 1분 1초가 급박하게 돌아가는 물류센터...)
- **질문형 문장 반복**: "~라면? ~라면?" 식으로 문제 상황을 독자가 상상하게 만든 뒤 "실제로 저희가 마주한 현실이었습니다"로 매듭짓는다.
- **수치/사실 기반 배경**: 회사·서비스의 성장이나 규모 변화로 시작해 "그런데 이 성장이 오히려 새로운 문제를 낳았다"로 전환한다.

도입부 마지막에는 이 글이 다루는 내용을 1~2문장으로 미리 요약하고, "비슷한 고민을 하는 개발팀에게 도움이 되길 바란다" 톤으로 짧게 마무리한다.

## 문체

- **존댓말**로 쓴다 ("~했습니다", "~입니다"). 기존 글의 반말 회고체("~했다")를 따라하지 않는다.
- 원인/장점/설계 원칙처럼 여러 개를 나열할 때는 **bold로 핵심 문구를 소제목처럼 강조**한 뒤 설명을 붙이는 리스트를 쓴다.
  ```
  - **데이터 단절로 인한 판단 불가**: 주문 단계에서 실시간 재고나 센터별 가동률 같은 물류 데이터를 즉시 확인하기 어려웠습니다.
  ```
- 이모지는 소량만 쓴다. 섹션 전환이나 감정이 실리는 지점에만 (🚀 🥲 🎯 👉 🔗 💡). 문장마다 붙이지 않는다.
- 💡 콜아웃 형태로 "프로세스 요약"이나 "참고"를 별도로 뽑아준다.
  ```
  💡 최적의 센터를 찾아가는 프로세스
  1단계. ...
  2단계. ...
  ```
- 표는 기술/대안 비교에 적극 활용한다. 표 없이 장단점을 줄글로 풀어쓰지 않는다.
- 정량적 결과는 문장 속에 흘려쓰지 않고, 섹션 끝에 강조해서 모아 보여준다 (가능하면 "지표 43.5% ↑" 같은 카드/불릿 형태).
- 결과가 수치로 안 나오는 항목은 지어내지 말고 정성적 변화(예: "불편하다는 피드백이 사라졌습니다")로 쓴다.
- 이 레포는 이미지 첨부 파이프라인이 없으므로 이미지·스크린샷은 넣지 않는다. 원본 올리브영 글의 이미지 자리는 표/코드블록/💡 박스로 대체한다.

## 마무리 작성법

마지막 섹션은 다음 요소를 순서대로 포함한다.

1. 종합 결과 수치 요약
2. 배운 점 / 교훈 (일반론 금지, 이 프로젝트에서만 나올 수 있는 구체적 교훈)
3. 남은 이슈 — 아직 못 푼 문제가 있다면, 무엇이 남았고 어떻게 푸는 게 옳다고 생각하는지 (없다면 생략)
4. (선택) 앞으로의 계획이나 로드맵
5. (선택) 독자에게 질문을 던지며 마무리 ("여러분의 서비스도 비슷한 고민을 앞두고 계신가요?")

## 체크리스트

새 글을 쓴 뒤 아래를 확인한다.

- [ ] 도입부가 설명이 아니라 장면/질문형 문장으로 시작하는가
- [ ] 존댓말로 일관되게 쓰였는가 (반말 섞이지 않았는가)
- [ ] 배경 섹션이 "결정(원칙)"과 "비교와 선택(구체적 기술)"을 건너뛰지 않고 순서대로 담고 있는가 — 원칙에서 곧장 기술명으로 점프하지 않았는가
- [ ] 방향 비교가 표로 한 번 정리되어 있는가
- [ ] 사이클(N차 문제)마다 어려움-해결이 드러나는가, 비교/트레이드오프를 매번 반복해서 늘어놓지 않았는가
- [ ] 원인/장점 나열에 bold 강조 리스트를 썼는가
- [ ] 결과가 마지막에 수치로 몰아서 정리되어 있는가 (지어낸 수치 없이)
- [ ] 마무리에 교훈이 구체적으로 들어가 있는가
- [ ] 남은 이슈가 있다면 숨기지 않고 마무리에 담았는가
- [ ] 비난·저격으로 읽힐 수 있는 표현이 없는가
- [ ] `lib/posts.ts`의 `POSTS`에 slug/title/description/date/tags를 추가했는가
