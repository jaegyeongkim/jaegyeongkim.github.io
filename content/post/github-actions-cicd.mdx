배포를 자동화하기 전에는 누군가 직접 터미널을 열어야 했다. `npm run build`를 실행하고, 결과물을 S3에 올리거나 서버에 SSH로 접속해서 명령어를 치는 식이었다. 앱이 2~3개일 때는 그냥 했는데, 10개가 되니까 이게 일이 됐다.

자동화를 미뤘던 이유는 "지금 당장 급한 게 아니라서"였다. 돌이켜보면 매 배포마다 수동으로 하는 시간과 실수할 가능성을 그냥 감수하고 있었던 거다. Turborepo로 모노레포 전환을 하면서 같이 정리하기로 했다.

## 두 가지 배포 대상

앱이 두 종류다.

- **React SPA** (어드민 앱 10개): 정적 파일이라 S3 + CloudFront로 충분하다
- **Next.js 앱** (랜딩 페이지 3개): SSR이 필요해서 Docker + ECR + EC2로 운영한다

배포 방식이 달라서 파이프라인도 분리했다.

## React SPA: S3 + CloudFront

빌드 결과물을 S3에 올리고 CloudFront로 서빙한다. 파이프라인에서 핵심은 **변경된 앱만 빌드하는 것**이다.

```yaml
- name: 빌드 (변경된 앱만)
  run: pnpm turbo build --filter=...[HEAD^1]
```

`--filter=...[HEAD^1]`은 이전 커밋과 비교해 변경된 패키지와 그에 의존하는 패키지만 빌드하라는 뜻이다. `packages/components`를 수정했는데 `kokkokAdmin`만 그걸 쓰고 있다면, `kokkokAdmin`만 다시 빌드된다. 나머지 9개는 건너뛴다.

```yaml
- name: S3 업로드
  run: |
    aws s3 sync apps/kokkokAdmin/build-prod-kokkok-admin/ \
      s3://${{ secrets.S3_BUCKET }}/kokkokAdmin \
      --delete

- name: CloudFront 캐시 무효화
  run: |
    aws cloudfront create-invalidation \
      --distribution-id ${{ secrets.CF_DISTRIBUTION_ID }} \
      --paths "/*"
```

`--delete` 플래그가 있어서 이전 빌드 파일이 S3에 남지 않는다. CloudFront 캐시 무효화는 배포 직후 사용자가 새 버전을 받을 수 있게 한다.

## Next.js: Docker + ECR + EC2

정적 파일이 아니라 서버가 필요하기 때문에 Docker 이미지로 배포한다.

```yaml
- name: Docker 이미지 빌드
  run: |
    docker build \
      -f apps/landing/Dockerfile \
      -t ${{ secrets.ECR_REGISTRY }}/landing:${{ github.sha }} \
      .

- name: ECR 푸시
  run: docker push ${{ secrets.ECR_REGISTRY }}/landing:${{ github.sha }}

- name: EC2 배포
  uses: appleboy/ssh-action@v1
  with:
    host: ${{ secrets.EC2_HOST }}
    username: ec2-user
    key: ${{ secrets.EC2_SSH_KEY }}
    script: |
      docker pull ${{ secrets.ECR_REGISTRY }}/landing:${{ github.sha }}
      docker stop landing || true
      docker run -d --name landing \
        -p 3000:3000 \
        --restart unless-stopped \
        ${{ secrets.ECR_REGISTRY }}/landing:${{ github.sha }}
```

이미지 태그로 `github.sha`를 쓴다. ECR에 이미지가 커밋 해시별로 쌓이기 때문에 "언제 배포된 어떤 이미지가 지금 올라가 있는지"를 추적할 수 있다.

## 롤백

React SPA는 코드를 `git revert`하고 파이프라인을 다시 돌리는 방식이다. 배포 자체가 수 분 안에 끝나서 현실적으로 가능하다.

Next.js는 ECR에 이미지가 쌓여 있어서 수동 롤백 워크플로우를 따로 만들었다.

```yaml
on:
  workflow_dispatch:
    inputs:
      image_tag:
        description: '롤백할 커밋 SHA'
        required: true
```

GitHub Actions UI에서 태그를 입력하고 실행하면 해당 버전으로 돌아간다.

## 파이프라인이 20개가 된 이유

앱 하나당 파이프라인이 하나가 아니다.

| 종류 | 개수 | 트리거 |
|---|---|---|
| PR 린트 + 타입 체크 | 1 | PR 오픈 |
| PR 테스트 | 1 | PR 오픈 |
| React 배포 dev | 5 | develop push |
| React 배포 prod | 5 | main push |
| Next.js 배포 dev | 3 | develop push |
| Next.js 배포 prod | 3 | main push |
| 롤백 | 2 | 수동 |

PR 단계에서 린트, 타입 체크, 테스트를 통과해야 merge할 수 있다. 배포는 브랜치 push에만 반응한다. 검증을 통과한 코드만 배포된다는 뜻이다.

---

자동화 전후로 달라진 게 단순히 편의성만은 아니었다. 배포가 두렵지 않아지면 배포 주기가 짧아진다. 예전에는 "이 정도면 모아서 한 번에 배포하자"는 말이 자연스러웠는데, 지금은 작은 단위로 자주 배포하는 게 당연해졌다.
