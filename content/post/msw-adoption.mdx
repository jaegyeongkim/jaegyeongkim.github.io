"서버는 아직인데, 화면은 언제 볼 수 있어요?"

신규 이벤트 앱 kokkok-event를 만들 때 마케팅팀에서 자주 받던 질문이었습니다. 친구 초대, 출석체크, 룰렛 세 가지 이벤트를 정해진 오픈일에 맞춰 열어야 했는데, API 명세는 노션 문서로만 존재했고 실제 서버는 아직 개발 중이었습니다. 화면 쪽 작업을 그대로 두면 서버가 나올 때까지 손을 놓고 기다려야 하는 상황이었습니다.

이런 일이 처음은 아니었습니다. 어드민 서비스 kokkok-admin을 새로 만들 때도 화면마다 같은 문제가 반복됐고, 그때마다 팀에서 택한 답은 MSW였습니다. 이번 kokkok-event도 같은 방식으로 풀기로 하고, 제가 이 앱의 MSW 세팅과 이벤트 3종의 mock 개발을 맡았습니다. 그 과정에서 마주친 문제와 해결 과정을 정리합니다. 비슷한 상황에 있는 개발팀에게 도움이 되길 바랍니다.

## 왜 화면부터 완성해야 했나

문제는 명확했습니다. 화면 개발이 API 서버의 존재 여부에 종속돼 있었습니다. 서버가 없으면 요청을 보낼 곳이 없고, 요청을 보낼 곳이 없으면 로딩 상태 이후의 화면을 확인할 방법이 없었습니다. 이벤트처럼 오픈일이 고정된 작업에서는 이 종속 관계 자체가 리스크였습니다.

그래서 세운 원칙은 "API 명세만 확정되면, 실제 서버 없이도 화면을 완성할 수 있는 계층을 프론트 쪽에 둔다"는 것이었습니다. 서버 개발과 화면 개발을 같은 명세를 기준으로 병렬로 진행하고, 서버가 준비되면 그 계층만 걷어내고 실제 API로 바꿔 끼우면 되는 구조를 만드는 게 목표였습니다.

이 계층을 무엇으로 만들지는 세 가지를 두고 비교했습니다.

| 방법 | 방식 | 한계 |
| --- | --- | --- |
| 컴포넌트에 mock 데이터 하드코딩 | API 호출부를 임시로 더미 값으로 대체 | 실제 API로 바꿀 때 코드를 다시 걷어내야 함, 네트워크 상태(로딩/에러)를 재현 못함 |
| axios interceptor로 응답 가로채기 | 요청 직전/직후에 가짜 응답을 끼워 넣음 | axios에 강하게 결합되어 fetch 기반 코드나 다른 요청 라이브러리에는 적용 안 됨 |
| MSW (Mock Service Worker) | Service Worker가 네트워크 레벨에서 요청을 가로챔 | 초기 세팅 비용이 있음 |

MSW를 선택한 이유는 네트워크 레이어에서 요청을 가로채기 때문에 앱 코드가 개발 환경인지 실제 환경인지 구분할 필요가 없다는 점이었습니다. 브라우저 개발자도구 Network 탭에는 실제 요청처럼 그대로 찍혔고, 나중에 서버가 준비되면 MSW를 끄는 것만으로 실제 API 호출로 전환됐습니다. 이미 사내 다른 레포에서 같은 방식으로 검증된 선례가 있었던 점도 컨벤션을 그대로 가져올 수 있어 유리했습니다.

kokkok-event는 새로 만드는 앱이었지만, 다른 서비스에서 이미 MSW를 도입해두었던 덕분에 mockServiceWorker.js 파일과 빌드 시 제외 설정 같은 공용 인프라는 그대로 가져다 쓸 수 있었습니다. 제가 새로 신경 써야 했던 부분은 이 앱에 필요한 mock 핸들러 자체, 그중에서도 친구 초대·출석체크·룰렛 세 이벤트의 실제 동작을 mock으로 재현하는 일이었습니다.

## 1차 문제: 룰렛이 응답값과 다른 칸에 멈췄다

친구 초대와 출석체크는 서버 응답을 그대로 화면에 옮기기만 하면 됐습니다. 하지만 룰렛은 달랐습니다. 서버가 "당첨된 상품이 무엇인지"를 응답으로 내려주면, 룰렛 애니메이션은 정확히 그 상품이 그려진 칸에서 멈춰야 했습니다.

처음 만든 mock 핸들러는 당첨 상품 정보만 반환했는데, 룰렛 컴포넌트 쪽 회전 로직과 기준이 어긋나 있었습니다. mock이 반환하는 "당첨 인덱스"와 룰렛이 실제로 멈추는 칸 계산 방식이 따로 놀았던 겁니다.

```ts
// mocks/roulette.ts
export const rouletteHandlers = [
  http.post("/api/roulette/spin", () => {
    // 서버가 내려줄 값과 동일한 형태로, 당첨 칸의 인덱스만 반환
    const resultIndex = pickWeightedIndex(ROULETTE_ITEMS);
    return HttpResponse.json({ resultIndex });
  }),
];
```

```ts
// useRouletteAnimation.ts
const targetAngle = calculateStopAngle({
  resultIndex, // mock/실서버 응답이 공통으로 내려주는 값
  itemCount: ROULETTE_ITEMS.length,
  extraSpins: 5, // 회전감을 위해 추가로 도는 바퀴 수
});
```

💡 mock과 애니메이션의 역할 분리
1단계. mock은 "몇 번 칸이 당첨인지"(`resultIndex`)만 결정한다.
2단계. 화면 쪽 애니메이션 로직이 그 값을 받아 "몇 바퀴를 돌아 몇 도에서 멈출지"를 계산한다.
3단계. 실서버로 전환해도 이 계약(`resultIndex`)만 같으면 애니메이션 코드는 그대로 동작한다.

해결 방향은 🎯 mock과 애니메이션 양쪽이 같은 값 하나(`resultIndex`)만 주고받도록 기준을 통일하는 것이었습니다. mock은 "몇 번 칸이 당첨인지"만 결정하고, 실제로 몇 바퀴를 돌아 몇 도에서 멈출지는 항상 화면 쪽 로직이 그 값을 기준으로 계산하도록 역할을 나눴습니다. 회전각을 계산하는 로직 자체는 여러 변수(추가 회전 수, 칸 개수, 시작 각도)가 얽혀 있어 복잡했는데, 이 부분은 AI 도구의 도움을 받아 계산식을 정리하고, 이후 팀원이 코드를 볼 때 각 단계를 따라갈 수 있도록 주석을 남겨두는 방식으로 처리했습니다. 로직 전체를 완벽히 손으로 이해하는 것보다, 흐름을 파악할 수 있게 남겨두는 쪽이 더 실용적이라고 판단했습니다.

## 2차 문제: 서버 스펙이 바뀔 때마다 mock이 조용히 낡아갔다

kokkok-admin 쪽에서는 다른 문제가 나타났습니다. 기능을 만들 때는 그 시점의 서버 명세에 맞춰 mock을 짰지만, 이후 서버가 관리하는 enum 값이 바뀌거나(예: `E_GIFT_TYPE_KEY_PAIR`, `PLATFORM_TYPE_KEY_PAIR`, `CUSTOMER_GRADE_KEY_PAIR` 등 다섯 종류) API 응답 형태가 수정돼도, 이미 짜둔 mock은 그대로 남아 있었습니다.

이게 위험했던 이유는, mock이 낡은 응답을 내려줘도 화면은 에러 없이 잘 동작하는 것처럼 보였기 때문입니다. 실제 서버 스펙과 어긋나 있다는 사실 자체를 알아챌 방법이 없었습니다. 화면 개발 단계에서는 문제가 없어 보이다가, 실제 서버와 연동하는 순간에야 값이 다르다는 게 드러나는 구조였습니다.

- **서버 enum이 바뀔 때**: 영향받는 mock 파일을 전수 조사해서, 관련된 화면과 mock을 한 PR에서 한꺼번에 반영했습니다. enum 다섯 종류가 동시에 바뀐 적도 있었는데, 그때마다 개별 화면 단위가 아니라 변경된 enum 기준으로 영향 범위를 먼저 추리고 작업했습니다.
- **mock 자체가 통째로 빠졌을 때**: 화면은 만들었는데 mock 코드 작성을 놓친 케이스가 누적돼 있던 걸 발견하고, 이런 누락 건만 모아 별도 PR로 일괄 점검·보완했습니다.

기능 단위로 mock을 만드는 방식은 화면을 빠르게 완성하는 데는 유리했지만, 서버 스펙이 바뀌는 순간을 mock이 스스로 알아채지 못한다는 한계가 분명했습니다. 결국 스펙 변경 이벤트(enum 변경, API 명세 수정)가 생길 때마다 mock도 같이 갱신 대상에 포함시키는 걸 작업 습관으로 삼는 수밖에 없었습니다.

## 종합 결과

- kokkok-event 이벤트 3종(친구 초대·출석체크·룰렛)을 실제 서버 없이 mock만으로 완성해 정해진 오픈 일정 안에 화면 개발을 마쳤습니다.
- kokkok-admin, kokkok-event, delivery, kokkokStore, moveAdmin 5개 앱에 MSW가 적용돼 있고, mock 핸들러 파일은 전체 70개(그중 kokkok-admin이 52개)입니다.
- 신규 앱 생성 스크립트에 MSW 기본 세팅이 템플릿으로 포함돼, 새 앱을 만들 때마다 mock 인프라를 처음부터 다시 구성할 필요가 없어졌습니다.
- 서버 enum 변경 다섯 건을 mock까지 한 번에 반영해, 스펙 불일치가 화면에서 조용히 방치되는 걸 막았습니다.

## 배운 점

MSW를 여러 앱, 여러 기능에 걸쳐 붙여보며 얻은 교훈은 이랬습니다.

- **화면 개발과 백엔드 개발은 API 명세만 확정되면 병렬로 갈 수 있습니다.** 서버가 없어도 명세만 있으면 화면을 끝까지 완성할 수 있었고, 실제 서버가 준비된 뒤에는 mock을 끄는 것만으로 전환이 끝났습니다. 오픈일이 정해진 작업일수록 이 병렬 구조가 일정 리스크를 줄여줬습니다.
- **mock도 코드입니다. 관리하지 않으면 낡습니다.** API가 바뀌어도 mock을 안 고치면 화면은 계속 멀쩡하게 동작하는 것처럼 보입니다. 그래서 오히려 사람이 일부러 챙기지 않으면 스펙 불일치를 아무도 모른 채 지나가기 쉽습니다. API 변경을 다룰 때는 mock 변경도 같은 무게로 체크리스트에 넣어야 한다는 걸 배웠습니다.
- **복잡한 로직은 혼자 다 짜지 않아도 됩니다.** 룰렛의 회전각 계산처럼 여러 값이 얽힌 로직은 AI 도구의 도움을 받아 빠르게 구현하고, 대신 주석으로 흐름을 남겨 팀원이 필요할 때 따라갈 수 있게 하는 것도 현실적인 선택이었습니다.

## 남은 이슈: mock과 실제 서버의 정합성 검증

지금은 서버 스펙이 바뀌었는지를 사람이 직접 찾아서 mock에 반영하고 있습니다. 이 과정 자체를 자동으로 검증할 장치는 아직 없습니다. API 응답 스키마와 mock 핸들러의 반환 타입을 같은 소스에서 비교해, 어긋나는 순간 CI에서 잡아주는 계약 테스트 같은 장치를 다음 단계로 고려하고 있습니다.

여러분의 팀도 백엔드가 준비되기 전에 화면부터 만들어야 하는 상황을 자주 겪으시나요? 그렇다면 mock을 임시방편으로만 쓰지 말고, API 명세가 바뀔 때마다 함께 갱신되는 코드로 다뤄보시길 권합니다.
