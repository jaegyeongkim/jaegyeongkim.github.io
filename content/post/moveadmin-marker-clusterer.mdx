지도에 마커 몇 개 찍는 건 프론트엔드 업무 중에서도 꽤 만만해 보이는 축에 속합니다. 저도 처음엔 그렇게 생각했습니다. position 찍고 map에 붙이면 끝 아닌가 하고요.

그 생각이 깨진 건 moveAdmin의 무브 모니터링 화면을 맡으면서였습니다. 이 화면은 운행 중인 드라이버 위치를 구글맵 위에 실시간으로 표시하는데, 운행 차량이 늘어나면 마커도 그만큼 늘어나는 구조였습니다. 문제는 마커 하나하나가 실제로는 DOM 노드였다는 점입니다. 드라이버가 몇십 명만 되어도 지도를 줌아웃하는 순간 마커들이 서로 겹치면서 클릭도 식별도 안 되는 상태가 됐고, 실시간 소켓으로 위치가 계속 갱신되다 보니 그 부하가 한 번으로 끝나지 않고 계속 반복됐습니다.

이 글은 그 문제를 마커 클러스터링으로 풀어간 과정과, 그 과정에서 떠밀리듯 먼저 해결해야 했던 선행 문제, 그리고 클러스터링을 붙이면서 뜻밖에 덕을 봤던 기존 설계에 대한 이야기입니다. 비슷한 고민을 하고 계신 분들께 도움이 되길 바랍니다.

## 마커가 곧 DOM 노드였던 구조

`ListBody.tsx`는 활성 드라이버 목록(`filteredDrivers`)이 바뀔 때마다 드라이버 수만큼 `google.maps.marker.AdvancedMarkerElement`를 새로 만들어 지도에 얹는 구조였습니다.

```ts
// ListBody.tsx (마커 생성 로직만 발췌)
const createVehicleMarker = (drivers) => {
  const newVehicleMarkers = drivers
    ?.map((driver) => {
      if (!driver?.id || !driver?.location || !googleMap) return;

      const vehicleStatus = getValueFromKeyPair({
        pair: MONITORING_STATUS_KEY_PAIR,
        value: driver.status,
        returnAsString: false,
      });
      const vehicleType = getValueFromKeyPair({
        pair: VEHICLE_TYPE_KEY_PAIR,
        value: driver.vehicleType,
        returnAsString: false,
      });
      if (!vehicleStatus || !vehicleType) return;

      const vehicleMarker = createGoogleMapMarker({
        map: googleMap,
        position: { lat: driver.location.lat, lng: driver.location.lng },
        src: CAR_DEFAULT[vehicleStatus][vehicleType],
        zIndex: zIndex.GOOGLE_MAP_DEFAULT_DRIVER_MARKER,
      });

      return { driverId: driver.id, marker: vehicleMarker };
    })
    .filter((marker) => marker !== undefined);

  setVehicleMarkers(newVehicleMarkers);
  return newVehicleMarkers;
};
```

드라이버 수가 늘어나는 만큼 이 배열이 그대로 커지고, 그만큼 지도 위 DOM 노드도 늘어납니다. 백엔드 페이지네이션이 있는 리스트라면 화면에 안 보이는 항목은 아예 안 그리면 되지만, 지도 위 마커는 "화면 안에 실제로 존재하는 차량"을 표시하는 것이라 임의로 잘라낼 수 없다는 게 이 문제의 까다로운 지점이었습니다.

원칙은 명확했습니다. **지도에 실제로 그려지는 노드 수를 드라이버 수가 아니라 밀집도에 비례하도록 묶어서 줄이자.** 이 원칙을 구현할 방법이 MarkerClusterer 하나만 있었던 건 아닙니다. 정식으로 문서화하고 비교한 절차는 아니었지만, 떠올려볼 수 있는 대안은 이런 것들이었습니다.

| 방식                     | 동작 방식                                                              | 걸리는 지점                                                                                                                            |
| ------------------------ | ---------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- |
| 뷰포트 밖 마커 제외      | 지도 이동·줌 이벤트마다 화면 밖 마커를 걸러내는 필터링 로직 추가       | 지도를 조작할 때마다 필터링 비용이 새로 발생하고, 정작 화면 안에서 겹치는 마커 문제는 해결되지 않음                                    |
| 캔버스/WebGL 레이어 전환 | 마커 레이어 전체를 캔버스 기반 렌더링으로 교체                         | 리스트 클릭 → 마커 하이라이트, 마커 클릭 → 리스트 스크롤 같은 기존 상호작용(`marker.addListener`, ref 기반 스크롤)을 전부 새로 짜야 함 |
| MarkerClusterer (그룹핑) | 기존 마커 배열을 그대로 넘기면 라이브러리가 몇 개를 어떻게 그릴지 계산 | 기존 마커 생성·이벤트 로직은 손대지 않고 그룹핑 레이어 하나만 얹으면 됨                                                                |

이 중 기존 마커 생성·이벤트 로직을 다시 짜지 않아도 된다는 점이 결정적이어서 `@googlemaps/markerclusterer`(MarkerClusterer)를 선택했습니다.

## 1차 문제: 클러스터링보다 먼저 풀어야 했던 마이그레이션

클러스터링을 붙이기로 마음먹었을 때 곧바로 라이브러리부터 설치할 수는 없었습니다. 당시 Google이 기존 `google.maps.Marker`를 종료(deprecate)한다고 공지했고, 그 대체제인 `google.maps.marker.AdvancedMarkerElement`는 DOM 기반이라 아이콘을 넘기는 방식부터 `mouseover`/`mouseout` 이벤트 처리까지 기존 코드와 호환되지 않았습니다.

- **해결**: `useActivateDriverMarker.ts`를 포함한 지도 관련 코드 전체를 `AdvancedMarkerElement`로 먼저 옮기는 작업을 진행했습니다. hover 효과도 마커 자체의 리스너가 아니라 아이콘 엘리먼트에 직접 거는 방식으로 바꿨습니다.

이건 성능을 위해 자발적으로 선택한 작업이 아니라 플랫폼이 강제한 마이그레이션이었습니다. 다만 결과적으로 이 작업이 끝나 있었기 때문에 클러스터링 라이브러리를 별다른 마찰 없이 얹을 수 있었습니다. DOM 기반 마커가 아니었다면, 클러스터러가 마커 콘텐츠를 그룹으로 묶어 대체 렌더링하는 방식 자체가 성립하지 않았을 것입니다.

## 2차 문제: 클러스터 아이콘이 한 번에 완성되지 않았다

처음 클러스터링을 붙였을 때는 커밋 메시지에도 남아있듯 "이미지 임시 적용", "라벨 스타일 임시 적용" 수준이었습니다. 구글 기본 클러스터 아이콘을 그대로 쓰다가, 이후 클러스터 안에 몇 대가 묶여있는지 보여주는 커스텀 렌더러로 교체했습니다.

```ts
// useActivateDriverMarker.ts (커스텀 렌더러 발췌)
const updateClusterMarker = (
  markers: google.maps.marker.AdvancedMarkerElement[],
): void => {
  if (clusteredMarker) {
    clusteredMarker.clearMarkers();
  }

  const newClusterMarker = new MarkerClusterer({
    map: googleMap,
    markers,
    algorithm: new SuperClusterAlgorithm({
      radius: 100,
      maxZoom: 17,
    }),
    renderer: {
      render: ({ position, markers }) => {
        const count = markers?.length ?? 0;

        return new google.maps.marker.AdvancedMarkerElement({
          position,
          content: (() => {
            const container = document.createElement("div");
            // 원형 배지 + 카운트 라벨을 직접 DOM으로 조립
            const label = document.createElement("span");
            label.innerText = count > 99 ? "99+" : `${count}`;
            container.appendChild(label);
            return container;
          })(),
          zIndex: zIndex.GOOGLE_MAP_DEFAULT_DRIVER_MARKER,
        });
      },
    },
  });

  setClusteredMarker(newClusterMarker);
};
```

이 커스텀 렌더러도 한 번에 확정된 게 아니라 QC를 거치며 계속 다듬어졌습니다. 클러스터 반경은 120 → 100으로 좁혔고, 배지 크기는 60px → 45px, 폰트는 18px → 15px로 줄었습니다. 그리고 밀집 지역에서 숫자가 세 자리, 네 자리로 늘어나며 배지를 뚫고 나가는 걸 막기 위해 `count > 99 ? "99+" : count`로 캡을 씌운 것도 이 단계였습니다.

클러스터링은 "붙이는 순간 끝"이 아니라, 실제 데이터 밀집도를 눈으로 확인하며 크기·반경을 맞춰가는 반복 작업에 가까웠습니다.

## 클러스터링을 붙이고도 손댈 필요가 없었던 것들

클러스터링을 도입하면서 걱정했던 부분이 하나 더 있었습니다. 리스트에서 드라이버 카드를 클릭하면 해당 마커가 강조되고 지도가 그 위치로 이동하는 기존 기능이, 마커 배열을 통째로 감싸는 클러스터링 레이어 밑에서도 그대로 동작할지였습니다.

막상 코드를 보니 걱정할 이유가 없었습니다. 이 화면의 마커 선택 로직은 애초부터 리액트 state를 거치지 않고, 이미 만들어진 `AdvancedMarkerElement` 인스턴스의 `content`/`zIndex`를 직접 mutate하는 방식으로 짜여 있었습니다. 클러스터링을 붙인 커밋에서도 이 부분은 건드리지 않았습니다.

```ts
// ListBody.tsx (마커 선택 처리 발췌 — 클러스터링 도입 이전부터 있던 코드)
currentVehicleMarker.zIndex = zIndex.GOOGLE_MAP_SELECT_DRIVER_MARKER;
currentVehicleMarker.content = createGoogleMapMarkerIcon(
  CAR_SELECT[vehicleStatus][vehicleType],
);
```

클러스터러 입장에서는 "묶여있는 마커 중 하나의 겉모습이 바뀐 것"일 뿐이라, 굳이 손대지 않아도 클러스터 전체가 다시 계산되지 않습니다. 만약 이 선택 로직이 리액트 state로 짜여 있었다면, 클러스터링을 붙이는 시점에 "선택 시 마커 배열 전체가 리렌더링되어 클러스터도 매번 다시 계산되는" 문제를 새로 풀어야 했을 겁니다. 기존 코드가 명령형으로 짜여 있었던 게, 결과적으로 클러스터링 레이어와 부딪히지 않은 이유였습니다.

💡 state로 갈지, mutate로 갈지 가르는 기준

- 리액트가 이 변경을 알아야 하위 컴포넌트도 함께 다시 그려져야 한다면 → state
- 서드파티 인스턴스(`AdvancedMarkerElement`)가 이미 들고 있는 값의 겉모습만 바꾸는 거라면 → mutate

정리 로직도 마찬가지였습니다. 이 화면은 소켓으로 드라이버 위치가 계속 갱신되기 때문에, `filteredDrivers`가 바뀔 때마다 마커를 새로 만들고 이전 마커는 `marker.map = null`로 지도에서 떼어내는 cleanup이 클러스터링 이전부터 이미 있었습니다. 클러스터링을 붙이면서 한 일은 이 기존 cleanup 옆에 클러스터 인스턴스를 정리하는 한 줄(`clusteredMarker.clearMarkers()`)을 처음부터 같이 추가한 것뿐이었습니다.

```ts
useEffect(() => {
  if (!googleMap || !filteredDrivers) return;
  const newVehicleMarkers = createVehicleMarker(filteredDrivers);

  return () => {
    if (clusteredMarker) {
      clusteredMarker.clearMarkers(); // 클러스터링 도입 커밋에서 함께 추가된 한 줄
    }
    newVehicleMarkers.forEach(({ marker }) => {
      if (!marker) return;
      marker.map = null; // 클러스터링 이전부터 있던 기존 cleanup
    });
  };
}, [filteredDrivers, googleMap, selectedStatus]);
```

클러스터러 인스턴스도 갱신 주기마다 새로 만들어지는 구조라, 정리해주지 않으면 화면엔 안 보여도 계속 누적됩니다. 기존 cleanup 패턴을 따라 같이 챙겨뒀기 때문에, 이 부분은 따로 사고가 난 적은 없습니다.

## 종합 결과

- 드라이버 수가 늘어나도 지도에 실제로 그려지는 노드 수는 드라이버 수가 아니라 밀집도에 따라 그룹핑되어 훨씬 적게 유지됩니다.
- 마커 겹침으로 인한 클릭·식별 문제가 해소되고, 클러스터 카운트 배지로 밀집 지역을 한눈에 파악할 수 있게 됐습니다.
- 마커 선택·정리 로직이 원래부터 mutate·cleanup 기반으로 짜여 있던 덕분에, 클러스터링을 얹고 나서도 리스트 선택 시 클러스터 전체가 재계산되지 않고 매끈하게 동작합니다.

다만 렌더링 시간이 정확히 몇 % 개선됐는지 같은 정량 수치는 커밋 히스토리에 남아있지 않습니다. 여러 차례 QC 라운드를 거치며 반경·크기를 계속 조정했다는 것 자체가, 체감 개선이 실제로 있었다는 정황 증거에 가깝습니다.

## 배운 점

- **서드파티 명령형 API를 감싸는 코드는, "리액트가 몰라도 되는 변경"을 state로 끌어들이지 않는 것만으로도 이후에 얹는 레이어(클러스터러 같은)와의 충돌을 미리 피할 수 있습니다.** 이 화면의 마커 선택 로직이 처음부터 mutate 기반이었던 덕분에, 클러스터링을 나중에 얹었을 때 리렌더링·재계산 문제를 새로 풀 필요가 없었습니다.
- **플랫폼이 강제하는 마이그레이션이 오히려 개선의 발판이 될 수 있습니다.** `google.maps.Marker` deprecation 대응이 없었다면, DOM 기반 클러스터링 자체를 붙이기 더 어려웠을 것입니다.
- **부가 레이어를 얹을 때는 갱신 주기에 맞춘 정리 로직을 처음부터 같이 설계해야 합니다.** 클러스터러 인스턴스도 기존 마커와 마찬가지로 매번 새로 만들어지는 구조라, 기존 cleanup 옆에 클러스터 정리(`clearMarkers()`)를 함께 추가해뒀습니다. 나중에 발견해서 고치는 것보다, 부가 레이어를 붙이는 시점에 기존 cleanup 패턴을 그대로 따라가는 게 비용이 훨씬 적었습니다.

## 남은 이슈: 감(感)에 의존한 튜닝 값과 검증 장치 부재

지도·클러스터링 로직은 앱(`apps/`) 코드라 이 레포 컨벤션상 자동화된 테스트 대상이 아닙니다(테스트는 `packages/**/__tests__`에만 존재). 그래서 클러스터 반경이나 배지 크기 같은 값은 실기기·실 데이터로 눈으로 확인하며 다듬는 수밖에 없었고, `radius: 100` / `maxZoom: 17` 같은 값도 여전히 감에 의존한 값입니다. 실제 드라이버 밀집 지역 데이터를 기반으로 한 값 재산정과, 회귀를 자동으로 잡아낼 수 있는 최소한의 검증 장치 마련이 남은 과제입니다.

여러분의 서비스도 실시간으로 갱신되는 대용량 데이터를 지도나 리스트 위에 그대로 그려야 하는 제약을 안고 계신가요? 그렇다면 화면에 보이는 노드 수 자체를 줄이는 방향을 먼저 검토해보시길 권합니다.
